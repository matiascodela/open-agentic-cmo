# Code of Conduct

## Our pledge

We as contributors, maintainers, and community members pledge to make participation in Open Agentic CMO a respectful, welcoming, and productive experience for everyone.

Open Agentic CMO is an open-source project focused on deterministic multi-agent workflows, structured artifacts, validation gates, and content operations.

We welcome contributions from people with different backgrounds, skill levels, experiences, and perspectives.

---

## Our standards

Examples of behavior that contributes to a positive environment include:

- Being respectful and constructive
- Giving and receiving feedback with care
- Assuming good intent
- Asking clarifying questions before criticizing
- Focusing on the work, not the person
- Respecting different experience levels
- Helping others understand the system
- Keeping discussions relevant and useful
- Documenting decisions clearly
- Prioritizing safety, reliability, and transparency

Examples of unacceptable behavior include:

- Harassment, intimidation, or personal attacks
- Insults, hostile comments, or dismissive language
- Discrimination of any kind
- Trolling or deliberately disruptive behavior
- Publishing someone else’s private information
- Sharing credentials, secrets, or private data
- Posting offensive, graphic, or inappropriate content
- Repeatedly derailing technical discussions
- Misrepresenting project behavior or capabilities
- Pressuring maintainers or contributors unfairly

---

## Project-specific expectations

Open Agentic CMO is a contract-driven system.

When participating in this project, please respect the design principles of the system:

- Agents should have clear responsibilities.
- Signals should be explicit.
- Artifacts should be structured.
- Validation gates should not be bypassed.
- Examples should be safe and synthetic.
- Security-sensitive data should never be shared publicly.
- Documentation should describe supported behavior clearly.
- Contributions should make the system more reliable, not more ambiguous.

Disagreements about architecture, agent behavior, schemas, or documentation are welcome when discussed respectfully and constructively.

---

## Scope

This Code of Conduct applies in all project spaces, including:

- GitHub Issues
- Pull requests
- Discussions
- Documentation reviews
- Code reviews
- Community chats
- Any other public or private project-related communication

It also applies when someone is representing the project publicly.

---

## Reporting issues

If you experience or witness unacceptable behavior, please report it to the project maintainers.

If the issue involves sensitive information, do not open a public issue.

Use a private reporting channel when available.

A report should include:

- What happened
- Where it happened
- Who was involved
- Any relevant links or screenshots
- Whether immediate action may be needed

Reports will be handled with care and discretion.

---

## Enforcement

Maintainers are responsible for clarifying and enforcing standards of acceptable behavior.

Possible enforcement actions include:

- Clarifying expectations
- Asking for edits to comments, issues, or pull requests
- Removing inappropriate content
- Temporarily limiting participation
- Permanently banning participants from project spaces

The goal of enforcement is to protect the community and maintain a constructive project environment.

---

## Good-faith technical disagreement

This project welcomes strong technical opinions.

It is acceptable to disagree about:

- Agent responsibilities
- Signal contracts
- Artifact schemas
- Validation rules
- Notion persistence models
- Testing strategy
- Documentation structure

It is not acceptable to turn technical disagreement into personal attacks.

Good disagreement should be specific, respectful, and focused on improving the project.

---

## Security and private data

Because this project may involve integrations with Notion, Telegram, and other external services, all contributors must be careful with sensitive information.

Do not post:

- API keys
- Tokens
- Database IDs
- Private URLs
- Customer data
- Internal logs
- Private screenshots
- Personal information

If you accidentally share sensitive information, notify maintainers and rotate exposed credentials immediately.

For security details, see `SECURITY.md`.
